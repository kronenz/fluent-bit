200노드(Spark·Trino·Airflow·Postgres·Monitoring)에서 fluent-bit(DaemonSet)이 CRI 컨테이너 로그를 수집·버퍼·가공해 OpenSearch 5노드(master+coord+data 일체형)로 적재하는 전체 흐름과 컴포넌트 간 상호작용(메타조회·버스트 흡수·역압·샤드 라우팅)을 한 장에.

2. 흐름 요약 (좌 → 우)

구간상호작용핵심
Workloads → CRI 로그각 Pod stdout/stderr → /var/log/containers/*.logCRI 포맷(partial 라인)
CRI → INPUT(tail)tail + filesystem 버퍼로 흡수, CRI multiline 재조립① 버스트 흡수
kubernetes 필터 ⇄ kubelet로컬 kubelet :10250에서 메타 조회(apiserver 우회)② apiserver 부하 소거
multiline/throttle/trim스택트레이스 그룹화 · 폭주 캡 · 노이즈/필드 드롭③④ doc·총량 감소
OUTPUT → Coordbulk index(generateID) ×200노드 → coordinating layer 수신⑤ 멱등
Coord → Data 샤드coord가 문서를 각 data 노드 primary/replica 샤드로 라우팅write pool 분산
역압(backpressure)포화 시 429/timeout → 재시도 → 버퍼 적재 → input pause유실 대신 흡수

3. 주의 / 함정 (실측 기반)

!Buffer_Size ≠ 배치/호출 크기 — OpenSearch 응답 읽는 버퍼. 줄여도 호출 안 늘고, 너무 작으면 could not flush. 크게/False 권장.
!keepLog:false 사각지대 — JSON 로그가 log를 잃어 log:error 검색서 누락 → 에러탐지를 msg/구조화 필드까지 확장.
imandatory 필터에 level/severity 금지 — 구조화 로그에만 존재 → 평문(klog/postgres)이 0건으로 사라짐.
