# Blameless Postmortem과 재발 방지 문화

> **TL;DR**: Blameless Postmortem은 개인이 아닌 시스템/프로세스의 결함을 찾아 재발을 방지하는 장애 검토 방식이다.
> 타임라인 재구성, 근본 원인 분석(5 Whys), 구체적 Action Items가 포스트모템의 세 가지 핵심 요소이다.
> 포스트모템의 가치는 문서 자체가 아니라, Action Items가 실행되고 시스템이 개선되는 것에 있다.

> **면접 빈출도**: ★★☆
> **예상 소요 시간**: 20min

---

## 핵심 개념

### Blameless Postmortem이란?

```
Blame 문화 (BAD):                    Blameless 문화 (GOOD):
"누가 잘못된 설정을 배포했나?"        "왜 잘못된 설정이 프로덕션에 도달했나?"
       │                                    │
       ▼                                    ▼
"A가 실수했다"                        "배포 전 설정 검증이 없었다"
       │                                    │
       ▼                                    ▼
A를 질책/처벌                         설정 검증 자동화 도입
       │                                    │
       ▼                                    ▼
팀원들이 실수를 숨김                  실수를 투명하게 공유
       │                                    │
       ▼                                    ▼
같은 실수 반복, 더 큰 장애            시스템이 점점 강건해짐
```

**Blameless의 핵심 전제**:
- 장애는 복잡한 시스템에서 불가피하게 발생한다 (Normal Accident Theory)
- 사람은 실수한다. 실수를 방지하는 것은 시스템의 책임이다
- 비난은 정보 공유를 억제하고, 같은 실수를 반복하게 만든다
- "이 상황에서 그 사람의 행동은 합리적이었다"는 전제로 시작한다

### 포스트모템 프로세스

```
장애 종료
    │
    ▼ (48~72시간 이내)
┌────────────────────────────┐
│ 1. 타임라인 초안 작성       │  ← IC/Ops Lead가 Slack 기록 기반으로
│    (사실만, 해석 없이)      │
└──────────┬─────────────────┘
           ▼
┌────────────────────────────┐
│ 2. 포스트모템 미팅 (60min)  │  ← 관련자 전원 참석
│    - 타임라인 리뷰 (15min)  │
│    - 근본 원인 분석 (25min) │
│    - Action Items (20min)   │
└──────────┬─────────────────┘
           ▼
┌────────────────────────────┐
│ 3. 문서 정리 + 공유         │  ← 전사 공유 (투명성)
│    Wiki/Notion/Confluence   │
└──────────┬─────────────────┘
           ▼
┌────────────────────────────┐
│ 4. Action Items 추적        │  ← JIRA 티켓, 주간 리뷰
│    (완료될 때까지)           │
└────────────────────────────┘
```

### 5 Whys 분석 기법

```
5 Whys의 핵심: 증상이 아닌 시스템/프로세스 원인에 도달해야 함

Good 5 Whys:                          Bad 5 Whys:
Why 서비스 다운?                       Why 서비스 다운?
→ OOM Kill 발생                        → 엔지니어가 실수함
Why OOM Kill?                          Why 실수?
→ 메모리 limit이 부족                  → 부주의
Why limit이 부족?                      Why 부주의?
→ 모델 크기 변경 후 미조정             → 피곤했음
Why 미조정?                            (여기서 멈춤 = 의미 없음)
→ 모델 변경 시 리소스 리뷰 프로세스 없음
Why 프로세스 없음?
→ 모델/인프라 팀 간 변경 알림 체계 부재

→ Action: 모델 변경 시 인프라 팀 자동 알림 + 리소스 리뷰 필수화
```

**5 Whys 함정 방지**:
```
1. 사람에서 멈추지 말 것
   BAD: "A가 테스트를 안 했다" (끝)
   GOOD: "테스트를 강제하는 CI 게이트가 없었다"

2. 단일 원인에 집착하지 말 것
   대부분의 장애는 여러 요인이 복합적으로 작용
   → Contributing Factors를 별도로 나열

3. 너무 넓게 가지 말 것
   "예산이 부족해서" → 실행 가능한 Action Item이 나올 수 없음
   적정 수준: 엔지니어링 팀이 실행 가능한 범위까지
```

### Action Items 품질 기준

```
GOOD Action Item:                      BAD Action Item:
├─ 구체적(Specific)                    ├─ 모호함
│  "transformers==4.36.2로 pin"        │  "테스트를 더 잘하자"
├─ 담당자 명시                         ├─ 담당자 없음
│  "@bob, 1/17까지"                    │  "팀에서 알아서"
├─ 측정 가능(Measurable)               ├─ 완료 기준 불명확
│  "CI에 모델 로딩 테스트 추가,        │  "모니터링 개선"
│   통과율 100%"                       │
├─ 시스템/프로세스 개선                 ├─ 개인 행동 변화
│  "pre-commit hook으로 pin 검증"      │  "다음부터 조심하자"
└─ 추적 가능                           └─ 추적 불가
   JIRA 티켓으로 관리                     구두 합의
```

### 포스트모템 메트릭

```
포스트모템 문화의 건강 지표:

1. 포스트모템 완료율
   = 작성된 포스트모템 / SEV1+SEV2 장애 수 × 100
   목표: 100%

2. Action Item 완료율
   = 완료된 AI / 전체 AI × 100
   목표: > 90% (3개월 이내)

3. 포스트모템 작성 시간
   = 장애 종료 → 포스트모템 문서 완성
   목표: < 72시간

4. 재발률
   = 동일 근본 원인 장애 / 전체 장애 × 100
   목표: < 10%

5. MTTR 추세
   = 월별 평균 MTTR
   목표: 지속적 감소
```

## 면접 Q&A

### Q: Blameless Postmortem이 왜 중요하고, 어떻게 진행하나?
**30초 답변**: Blameless Postmortem은 개인을 비난하지 않고 시스템/프로세스의 결함을 찾아 개선하는 장애 검토입니다. 비난하면 사람들이 실수를 숨기고 같은 장애가 반복됩니다. 타임라인 작성 → 5 Whys 근본 원인 분석 → Action Items 도출 → 추적의 순서로 진행합니다.

**2분 답변**: Blameless Postmortem의 핵심 전제는 "이 상황에서 그 사람의 행동은 당시 갖고 있던 정보를 기반으로 합리적이었다"입니다. 예를 들어, 테스트 없이 배포한 것은 "게으름"이 아니라 "테스트를 강제하는 CI 게이트가 없었기 때문"입니다. 진행 순서: (1) 장애 종료 후 48~72시간 이내에 미팅을 잡습니다. 너무 늦으면 기억이 흐려집니다. (2) IC가 타임라인 초안을 작성하고, 관련자가 보완합니다. Slack 로그, 알림 히스토리, Git 커밋 기록이 객관적 자료입니다. (3) 5 Whys로 근본 원인을 파고들되, 사람이 아닌 시스템/프로세스에서 답을 찾습니다. (4) 구체적이고 담당자가 있는 Action Items를 도출합니다. "조심하자"가 아니라 "CI에 검증 단계를 추가한다(@bob, 1/24)"처럼. (5) Action Items를 JIRA로 추적하고, 완료되지 않으면 매주 리뷰합니다. 포스트모템의 가치는 문서가 아니라 Action Items의 실행에 있습니다. Action Item 완료율이 낮으면 포스트모템 문화가 형식적이 되어 의미를 잃습니다.

**경험 연결**: 온프레미스 인프라에서 장애 보고서를 작성할 때 "원인: 담당자 부주의"로 끝나는 경우가 많았습니다. 그 결과 같은 유형의 장애가 반복되었습니다. Blameless 접근으로 전환하면 "부주의"가 아닌 "자동 검증 부재"를 원인으로 보고, 자동화로 해결합니다.

**주의**: "Blameless는 책임 회피가 아니다"를 명확히. 시스템을 개선하는 것이지, 아무도 책임지지 않는 것이 아님.

### Q: 5 Whys 분석의 올바른 진행 방법과 함정은?
**30초 답변**: 5 Whys는 "왜?"를 반복하여 표면적 원인에서 근본 원인까지 도달하는 기법입니다. 핵심 함정은 세 가지: 사람에서 멈추기, 단일 원인 집착, 너무 추상적 원인 도출입니다. 항상 "시스템/프로세스에서 무엇이 이것을 허용했는가?"를 물어야 합니다.

**2분 답변**: 5 Whys는 Toyota Production System에서 유래한 간단하지만 강력한 분석 기법입니다. 올바른 진행 방법: 첫째, 사실에서 출발합니다. "서비스가 다운되었다" → "OOM Kill이 발생했다" → "메모리 limit이 부족했다"처럼 검증 가능한 사실의 체인. 둘째, 각 "왜?"에서 여러 답이 가능하면 분기하여 탐색합니다. 하나의 원인만 있는 경우는 드뭅니다. 셋째, "엔지니어링 팀이 실행 가능한 Action Item"이 나오는 수준에서 멈춥니다. 주요 함정: (1) 사람에서 멈춤 - "A가 실수했다"가 아니라 "실수를 방지하는 시스템이 없었다". (2) 한 가닥만 추적 - 실제 장애는 여러 요인의 조합. Contributing Factors를 별도 목록으로 관리합니다. (3) 추상화 과잉 - "예산이 부족했다", "시간이 없었다"는 실행 가능한 조치가 아닙니다. (4) 확증 편향 - 미리 결론을 정해놓고 5 Whys를 맞추려 하는 것. 데이터(로그, 메트릭, 타임라인)로 각 단계를 뒷받침해야 합니다.

**경험 연결**: 인프라 장애 분석에서 "원인: 하드웨어 고장"으로 끝나는 경우가 많았는데, 5 Whys를 적용하면 "왜 단일 하드웨어 고장이 서비스 중단으로 이어졌나?" → "이중화가 없었다" → "이중화 구축 계획이 없었다" → "용량 계획 프로세스 부재"까지 도달할 수 있습니다.

**주의**: 5 Whys가 유일한 분석 방법은 아님. 복잡한 시스템 장애에는 Fishbone Diagram, Fault Tree Analysis 등도 활용.

### Q: 포스트모템의 Action Items가 실행되지 않는 문제를 어떻게 해결하나?
**30초 답변**: 세 가지 방법입니다. 첫째, Action Items를 반드시 JIRA 티켓으로 생성하여 추적합니다. 둘째, 주간 리뷰에서 진행 상황을 확인합니다. 셋째, P1 Action Items는 스프린트에 반드시 포함하여 우선순위를 보장합니다.

**2분 답변**: Action Item 미실행은 포스트모템 문화의 가장 큰 위협입니다. "포스트모템은 했지만 아무것도 변하지 않았다"면 팀의 신뢰가 무너집니다. 해결 전략: 첫째, **즉시 티켓화**: 포스트모템 미팅 끝나기 전에 모든 Action Items를 JIRA/Linear에 생성합니다. 담당자, 마감일, 우선순위를 미팅에서 합의합니다. 둘째, **스프린트 통합**: P1 Action Items는 다음 스프린트에 반드시 포함합니다. Product Owner와 사전 합의하여 "안정성 작업 예산"을 확보합니다(스프린트 용량의 20%). 셋째, **주간 리뷰**: 매주 팀 미팅에서 열린 Action Items를 리뷰합니다. 지연 사유가 있으면 재조정하되, 삭제는 하지 않습니다. 넷째, **메트릭 추적**: Action Item 완료율, 평균 완료 기간을 분기별로 측정합니다. 목표는 90% 이상 완료(3개월 이내). 다섯째, **경영진 지원**: 엔지니어링 리더가 포스트모템 Action Items를 정기적으로 리뷰하고, 리소스 부족 시 추가 지원을 결정합니다.

**경험 연결**: 인프라 장애 보고서의 개선 사항이 이행되지 않아, 같은 유형의 장애가 반복된 경험이 있습니다. 핵심은 "누가 언제까지"를 명확히 하고, 정기적으로 추적하는 것입니다.

**주의**: Action Items 전부가 P1일 수는 없다. 우선순위를 현실적으로 배정하되, P1은 반드시 지키는 문화가 중요.

## Allganize 맥락

- **스타트업 문화**: 소규모 팀에서 포스트모템은 전원이 참여하여 학습하는 기회. 비난 문화가 자리잡으면 빠르게 팀이 와해됨
- **AI 서비스 고유 장애**: 모델 할루시네이션, 추론 품질 저하 등 전통적 인프라 장애와 다른 유형의 포스트모템 필요
- **빠른 성장**: 서비스 성장 과정에서 장애는 불가피. 포스트모템을 통해 시스템이 점진적으로 강건해지는 것이 핵심
- **멀티클라우드 장애**: AWS/Azure 동시 운영에서 발생하는 복합 장애의 근본 원인 분석에 5 Whys 활용
- **고객 신뢰**: B2B 고객에게 장애 후 포스트모템 요약을 공유하면 투명성과 전문성을 어필할 수 있음
- **Game Day**: 정기적인 Chaos Engineering으로 "포스트모템 없는 학습"을 실현
