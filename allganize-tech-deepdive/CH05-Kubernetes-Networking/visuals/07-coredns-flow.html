ndots:5 Problem (ndots 문제와 불필요한 DNS 쿼리)

Example: Pod queries "api.example.com" (dots=2, < ndots:5)

점이 5개 미만이므로 search domain을 먼저 모두 시도한 후에야 원래 도메인 쿼리

1api.example.com.default.svc.cluster.localNXDOMAIN
2api.example.com.svc.cluster.localNXDOMAIN
3api.example.com.cluster.localNXDOMAIN
4api.example.com.93.184.216.34
Total: 4 DNS queries for 1 external domain! (3 unnecessary queries)

Solution 1: FQDN with trailing dot

api.example.com. # trailing dot = absolute name

끝에 점을 붙이면 search domain 추가 없이 바로 쿼리

Solution 2: Lower ndots

dnsConfig: options: - name: ndots value: "2"

ndots를 낮추면 외부 도메인 쿼리 횟수 감소 (trade-off 주의)

1ndots:5 기본값: 점이 5개 미만인 이름은 상대 이름으로 취급
ndots:5 default: names with < 5 dots treated as relative
2외부 도메인 접근 시 3개의 불필요한 DNS 쿼리 발생 (성능 저하)
External domain access causes 3 unnecessary DNS queries
3해결: FQDN 끝에 점(.) 추가하거나 Pod의 dnsConfig에서 ndots 낮춤
Fix: trailing dot for FQDN or lower ndots in pod dnsConfig

Cache Hit vs Cache Miss (캐시 히트 vs 미스)

Cache HIT (캐시 히트)

이전에 조회한 레코드가 TTL 내에 있음

1Pod → CoreDNS query
2CoreDNS cache lookup
3Cache HIT! Return cached result
4Response: ~0.5ms latency
No upstream query needed. Fast response.

Cache MISS (캐시 미스)

캐시에 없거나 TTL이 만료됨

1Pod → CoreDNS query
2CoreDNS cache lookup
3Cache MISS! Forward to upstream
4Upstream DNS resolves
5Cache result, return to pod
6Response: ~5-50ms latency
1CoreDNS 캐시 플러그인: 성공/실패 응답을 TTL 동안 캐시
CoreDNS cache plugin: caches success/failure responses for TTL
2Cache Hit: 0.5ms 이내 응답, upstream 쿼리 불필요
Cache Hit: sub-millisecond response, no upstream query
3Cache Miss: upstream DNS 쿼리 필요, 5-50ms 지연 발생
Cache Miss: upstream query needed, 5-50ms additional latency

NodeLocal DNSCache Optimization (노드 로컬 DNS 캐시)

Without NodeLocal DNS

1Pod DNS query
2iptables DNAT to CoreDNS
3Network hop to CoreDNS Pod
4CoreDNS processes query
5Conntrack race condition risk
Issues: conntrack race (UDP), network hop latency, CoreDNS overload

With NodeLocal DNSCache

1Pod DNS query
2Local cache (169.254.20.10)
3Cache HIT → instant response
4Cache MISS → TCP to CoreDNS
Benefits: no conntrack (TCP), no network hop, reduced CoreDNS load

Pod

NodeLocal DNSCache
DaemonSet on each node
169.254.20.10 (link-local)

Upstream DNS
External resolver
1NodeLocal DNSCache: 각 노드에 DaemonSet으로 로컬 DNS 캐시 배포
NodeLocal DNSCache: deploys local DNS cache on every node as DaemonSet
2Pod → 169.254.20.10 (link-local) → 로컬 캐시로 직접 쿼리 (no iptables)
Pod queries link-local IP directly, bypassing iptables/conntrack
3캐시 미스 시 TCP로 CoreDNS 쿼리 → UDP conntrack race condition 해결
Cache miss uses TCP to CoreDNS, solving UDP conntrack race
4CoreDNS 부하 감소 + DNS 응답 지연 시간 대폭 감소
Reduces CoreDNS load and DNS response latency significantly
