# 03. 장점과 단점 — 시스템 전체를 보는 눈 & DevOps 경력 피벗

> **TL;DR**
> - 장점: **시스템 전체를 조망하는 역량**. 물리 서버부터 서비스 배포까지 한 사람이 그림을 그릴 수 있다.
> - 단점: 순수 DevOps 타이틀 경력은 짧지만, **10년 인프라 경험이 DevOps의 본질과 정확히 겹친다**.
> - 단점 답변의 핵심은 "인정 → 피벗 → 구체적 보완 노력"의 3단 구조다.

> **면접 빈출도**: ★★★
> **예상 소요 시간**: 15min

---

## 장점: "시스템 전체를 보는 눈"

### 스크립트

```
제 가장 큰 강점은 인프라의 전체 그림을 볼 수 있다는 것입니다.

예를 들어, 서비스 응답이 느리다는 이슈가 발생하면
대부분 애플리케이션 레벨에서 원인을 찾기 시작합니다.
하지만 저는 물리 서버의 NUMA 토폴로지부터
네트워크 경로, OS 커널 파라미터, 컨테이너 리소스 제한,
그리고 애플리케이션 설정까지 전체 스택을 체계적으로 점검합니다.

실제로 이전 프로젝트에서 GPU 서버의 추론 성능이 기대치의 60%밖에 나오지 않는
이슈를 분석한 적이 있습니다.
결국 원인은 NUMA 노드 간 메모리 접근 패턴과
PCIe 토폴로지 불일치였는데,
이건 OS와 하드웨어 레이어를 동시에 이해하지 못하면 찾기 어려운 문제였습니다.

DevOps 엔지니어로서 이런 풀스택 시야는,
특히 AI 서비스처럼 하드웨어 의존도가 높은 환경에서
큰 차별점이 될 수 있다고 생각합니다.
```

### 장점의 근거 (면접관이 파고들 때)

| 질문 | 답변 포인트 |
|---|---|
| "구체적으로 어떤 레이어를 다뤄봤나요?" | 물리(서버/스토리지/네트워크) → OS(커널/시스템) → 가상화(VM/컨테이너) → 오케스트레이션(K8s) → 배포(CI/CD) |
| "그게 DevOps에서 왜 중요한가요?" | 장애 원인의 **70%는 인프라 레이어에 있다**. 애플리케이션만 보는 DevOps는 절반만 볼 수 있다 |
| "남들과 뭐가 다른가요?" | 대부분의 DevOps 엔지니어는 **클라우드 위에서 시작**. 저는 **클라우드 아래도 안다** |

## 단점: "순수 DevOps 경력은 짧지만..."

### 안티패턴 — 이렇게 말하면 안 된다

```
(X) "단점은 DevOps 경험이 없다는 것입니다."
    → 자기 부정. 면접관이 "그럼 왜 지원했어?"라고 생각한다.

(X) "단점은 완벽주의입니다."
    → 진부한 클리셰. 면접관이 내심 실망한다.

(X) "단점은 없다고 생각합니다."
    → 자기 인식 부족으로 평가된다.
```

### 올바른 구조: 인정 → 피벗 → 보완

```
솔직히 말씀드리면,
"DevOps Engineer"라는 타이틀로 일한 경력은 짧습니다.

하지만 돌이켜보면, 제가 해온 일의 본질은
DevOps가 추구하는 것과 상당 부분 겹칩니다.

서버 프로비저닝 자동화를 위해 Ansible 스크립트를 작성했고,
배포 프로세스를 표준화하기 위해 문서화와 스크립팅을 반복했고,
장애 발생 시 원인 분석부터 재발 방지까지 주도했습니다.

다만, Terraform이나 ArgoCD 같은 모던 DevOps 도구들에 대해서는
실무 경험보다는 학습과 개인 프로젝트 수준이라는 점은 인정합니다.

그래서 지금 이 부분을 집중적으로 보완하고 있습니다.
개인 환경에서 Kubernetes 클러스터를 구축하고,
Terraform으로 인프라를 코드로 관리하는 연습을 하고 있고,
GitHub Actions로 CI/CD 파이프라인을 직접 만들어보고 있습니다.

경험의 깊이는 시간이 해결해줄 거라 생각하고,
기반이 되는 인프라 이해도는 이미 충분하다고 자신합니다.
```

### 피벗 테이블: 기존 경험 → DevOps 매핑

| 기존에 한 일 | DevOps 매핑 |
|---|---|
| Ansible로 서버 프로비저닝 자동화 | **IaC (Infrastructure as Code)** |
| 배포 절차 문서화 + 스크립팅 | **CI/CD 파이프라인 설계** |
| 장애 원인 분석 + 재발 방지 | **SRE / 인시던트 관리** |
| 폐쇄망 패키지 의존성 관리 | **컨테이너 이미지 빌드 최적화** |
| 서버 성능 튜닝 | **옵저버빌리티 + 용량 계획** |
| 다양한 고객 환경 대응 | **멀티 환경 배포 전략** |

## 추가 장단점 준비 (2번째 질문 대비)

### 추가 장점: 문제 해결 끈기

```
한 가지 더 말씀드리자면, 저는 문제를 끝까지 파고드는 편입니다.
"일단 재시작하면 되니까"가 아니라,
"왜 이게 발생했는지" 근본 원인을 찾을 때까지 놓지 않습니다.

폐쇄망 환경에서 일하다 보면 구글링이 안 되거든요.
매뉴얼과 소스 코드, 시스템 로그만으로 문제를 해결해야 하는 환경에서
이 습관이 자연스럽게 만들어졌습니다.
```

### 추가 단점: 혼자 해결하려는 경향

```
단점을 하나 더 말씀드리면,
문제를 혼자 해결하려는 경향이 있습니다.
끝까지 파고드는 게 장점이기도 하지만,
때로는 빨리 도움을 요청하는 게 팀 전체로는 더 효율적이잖아요.

이 부분을 인식하고 있고,
최근에는 "30분 룰"을 적용하고 있습니다.
30분 안에 실마리가 안 보이면 동료에게 공유하고 함께 해결하는 방식으로요.
```
