2. Runtime Comparison 런타임 비교

Docker (dockershim) K8s 1.24+ 제거

  • 가장 널리 알려진 컨테이너 런타임
  • dockershim은 Kubernetes 1.24에서 제거
  • Docker Desktop은 개발 환경에서 여전히 유용
  • containerd를 내부적으로 사용
kubelet → dockershim → dockerd → containerd → runc → container

containerd (권장 / Recommended)

  • CNCF graduated 프로젝트
  • Kubernetes 기본 런타임
  • Docker보다 가볍고 직접 CRI 지원
  • 이미지 관리, 스냅샷 기능 내장
kubelet → CRI plugin → containerd → runc → container

CRI-O

  • Kubernetes 전용 경량 런타임
  • Red Hat / OpenShift 기본 런타임
  • OCI 호환 이미지만 지원
  • 최소한의 기능만 포함 (빌드 기능 없음)
kubelet → CRI-O → runc/crun → container
3. OCI Runtime Spec Flow OCI 런타임 스펙 흐름

Step 1: config.json 생성

{
  "ociVersion": "1.0.2",
  "process": {
    "args": ["/bin/sh"],
    "env": ["PATH=/usr/bin"],
    "cwd": "/"
  },
  "root": {
    "path": "rootfs",
    "readonly": true
  },
  "linux": {
    "namespaces": [
      {"type":"pid"},
      {"type":"network"},
      {"type":"mount"}
    ]
  }
}
OCI 번들: config.json + rootfs 디렉토리

Step 2: runc create

$ runc create mycontainer

# 내부 동작:
1. config.json 파싱
2. namespace 생성 (pid, net, mnt...)
3. cgroup 설정
4. rootfs 마운트
5. seccomp 필터 적용
6. 컨테이너 프로세스 준비
   (아직 실행하지 않음)

→ Status: created
컨테이너 생성 - 아직 프로세스 미실행 상태

Step 3: runc start

$ runc start mycontainer

# 내부 동작:
1. init 프로세스 시작
2. prestart hooks 실행
3. 사용자 프로세스 exec
4. poststart hooks 실행

→ Status: running

$ runc list
ID            STATUS
mycontainer   running
컨테이너 실행 - PID 1 프로세스 시작
